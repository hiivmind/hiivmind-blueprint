# Prompt Caching Research for Blueprint-Lib

> **Purpose:** Document how Anthropic's prompt caching works and how it applies to both Claude Code plugin usage and future API-based agent development within the blueprint framework.
>
> **Last updated:** February 2026
>
> **Sources:** [Anthropic Prompt Caching Docs](https://platform.claude.com/docs/en/docs/build-with-claude/prompt-caching), [Anthropic Pricing Docs](https://platform.claude.com/docs/en/docs/about-claude/pricing), [Anthropic Blog](https://www.claude.com/blog/prompt-caching)

## 1. Overview

The blueprint framework is YAML-heavy. Each Claude Code session or API-based agent invocation pays a token cost to load framework context: the engine entrypoint, pattern files, templates, and skill definitions. Prompt caching allows the system to store the KV-cache (key-value cache from the transformer attention mechanism) for prompt prefixes, so identical context does not need to be reprocessed on every call.

**Key benefits for blueprint-lib:**
- **Cost reduction:** Cache reads cost 90% less than base input tokens
- **Latency reduction:** Up to 85% faster time-to-first-token for long prompts
- **More context, same cost:** Can include richer framework context (full pattern files, more examples) without proportional cost increase
- **Rate limit relief:** Cache hits are not deducted against rate limits

## 2. How It Works

### Request Flow

1. The system checks if a prompt prefix, up to a specified cache breakpoint, is already cached from a recent query.
2. If found, it uses the cached KV-cache, skipping recomputation for that prefix.
3. Otherwise, it processes the full prompt and caches the prefix once the response begins.

### Exact Prefix Matching

Cache hits require **100% identical** prompt segments, including all text and images up to and including the block marked with `cache_control`. The cache key is a cryptographic hash of the prompts up to the cache control point. A single character difference breaks the cache.

### Cache Hierarchy

Cache prefixes are created in this fixed order:

```
tools → system → messages
```

Each level builds on the previous ones. This means:
- If `tools` changes, all three levels are invalidated.
- If `system` changes, `tools` cache survives but `system` and `messages` are invalidated.
- If only `messages` change, both `tools` and `system` caches survive.

### Cumulative Cache Keys

When you mark a block with `cache_control`, the cache hash key is generated by hashing **all previous blocks** in the conversation sequentially. The cache for block N depends on the content of blocks 1 through N.

### Backward Sequential Checking

The system checks for cache hits by working **backwards** from your explicit breakpoint, checking each previous block in reverse order. This finds the longest possible cache hit automatically.

### Output Unaffected

Prompt caching has **no effect** on output token generation. The response is identical to what you would get without caching.

## 3. Key Parameters

### TTL (Time To Live)

| TTL | Default? | Refresh Behavior | Cost |
|-----|----------|-----------------|------|
| 5 minutes (`"5m"`) | Yes | Refreshed for free on each cache hit | 1.25x base input |
| 1 hour (`"1h"`) | No | Refreshed for free on each cache hit | 2x base input |

The cache cannot be manually cleared. Entries expire after the TTL elapses without a hit.

### Minimum Token Thresholds

Prompts shorter than these thresholds cannot be cached, even if marked with `cache_control`:

| Model | Minimum Cacheable Tokens |
|-------|------------------------:|
| Claude Opus 4.6, Opus 4.5 | 4,096 |
| Claude Haiku 4.5 | 4,096 |
| Claude Sonnet 4.5, Opus 4.1, Opus 4, Sonnet 4, Sonnet 3.7 | 1,024 |
| Claude Haiku 3.5, Haiku 3 | 2,048 |

### Maximum Breakpoints

Up to **4 cache breakpoints** (blocks with `cache_control`) per request.

Breakpoints themselves are free — you only pay for the content that gets cached or read.

### 20-Block Lookback Window

The system checks up to **20 blocks** before each explicit `cache_control` breakpoint. If a change occurred more than 20 blocks before the breakpoint, no cache hit occurs for the content before that change — unless you place an additional explicit breakpoint closer to it.

**Example:** 30 content blocks, `cache_control` on block 30:

| Scenario | What happens |
|----------|-------------|
| Block 31 added, nothing else changed | Cache hit at block 30. Only block 31 processed. |
| Block 25 modified | Backward check finds mismatch at 25, match at 24. Cache hit at block 24. Blocks 25-30 reprocessed. |
| Block 5 modified | Backward check from 30 reaches block 11 (20 checks). Stops. No cache hit. **All blocks reprocessed.** |
| Block 5 modified + breakpoint on block 5 | System checks from breakpoint at 5 → no match → checks block 4 → match. Cache hit at block 4. |

## 5. Implementation: Claude Code Plugin

### How Caching Works Automatically

When using Claude Code (the CLI), prompt caching is handled **automatically** by the runtime. Plugin authors do not need to add `cache_control` markers — Claude Code manages breakpoint placement internally.

**What Claude Code does:**
- Places `cache_control` breakpoints on the system prompt (including CLAUDE.md contents, tool definitions, project context)
- Places a breakpoint at the end of conversation history for incremental caching in multi-turn sessions
- During agentic tool-use loops, thinking blocks get cached alongside other content when tool results are passed back
- The conversation grows incrementally, and each turn's context is cached for the next

**What plugin authors can control:**

| Lever | Description |
|-------|-------------|
| **Content stability** | Keep system instructions, patterns, and templates consistent across calls to maximize cache hits |
| **Progressive disclosure** | Only load context that the current phase actually needs, rather than loading everything upfront |
| **Content ordering** | Put the most stable content first (engine entrypoint, core patterns), variable content last |
| **Content size** | Must exceed model's minimum threshold (1,024–4,096 tokens) for caching to activate |

**What plugin authors cannot control:**

- Breakpoint placement (Claude Code decides)
- TTL selection (Claude Code uses defaults)
- Number of breakpoints (Claude Code decides)
- Cache key generation (automatic, based on content hash)

### Progressive Disclosure as the Primary Optimization

Since we can't control breakpoints, the main optimization lever is **what content we load and when**:

```
Phase 1 (Discovery):    Engine entrypoint only (~2K tokens)
Phase 2 (Analysis):     + relevant pattern file (~5K tokens)
Phase 3 (Generation):   + template file (~3K tokens)
Phase 4 (Validation):   + validation rules (~4K tokens)
```

Each phase's system context builds on the previous phase's cached prefix. If the engine entrypoint doesn't change between phases, it stays cached.

### Blueprint-Lib Implications

| Framework layer | Token estimate | Stability | Caching benefit |
|----------------|---------------:|-----------|-----------------|
| Engine entrypoint | ~2,000 | Very high (changes rarely) | Excellent — cached across entire session |
| lib/patterns/ | ~3,000–8,000 each | High (changes per release) | Good — cached within a session |
| templates/ | ~1,000–3,000 each | High | Good — cached within a session |
| Skill SKILL.md | ~2,000–5,000 | Medium (changes during authoring) | Moderate — breaks cache during development |
| User conversation | Variable | Low | Incrementally cached per turn |

## 6. Implementation: API-Based Agent

For future API-based agents built on blueprint-lib, you have **explicit control** over caching.

### Python Example: Caching Framework Context

```python
import anthropic

client = anthropic.Anthropic()

# Load framework context once
engine_entrypoint = open("engine_entrypoint.md").read()
pattern_content = open("lib/patterns/workflow-generation.md").read()
template_content = open("templates/SKILL.md.template").read()

response = client.messages.create(
    model="claude-sonnet-4-5-20250929",
    max_tokens=4096,
    system=[
        {
            "type": "text",
            "text": engine_entrypoint,
            "cache_control": {"type": "ephemeral"}  # Breakpoint 1: engine
        },
        {
            "type": "text",
            "text": pattern_content,
            "cache_control": {"type": "ephemeral"}  # Breakpoint 2: patterns
        },
        {
            "type": "text",
            "text": template_content,
            "cache_control": {"type": "ephemeral"}  # Breakpoint 3: templates
        },
    ],
    messages=[
        {
            "role": "user",
            "content": [
                {
                    "type": "text",
                    "text": "Generate a new skill for plugin discovery.",
                    "cache_control": {"type": "ephemeral"}  # Breakpoint 4: conversation
                }
            ]
        }
    ],
)

# Check cache performance
usage = response.usage
print(f"Cache write: {usage.cache_creation_input_tokens} tokens")
print(f"Cache read:  {usage.cache_read_input_tokens} tokens")
print(f"Uncached:    {usage.input_tokens} tokens")
```

### TypeScript Example: Caching Tool Definitions + System Prompt

```typescript
import Anthropic from '@anthropic-ai/sdk';
import { readFileSync } from 'fs';

const client = new Anthropic();

const engineEntrypoint = readFileSync('engine_entrypoint.md', 'utf-8');
const patternContent = readFileSync('lib/patterns/workflow-generation.md', 'utf-8');

const blueprintTools = [
  {
    name: "read_skill",
    description: "Read a skill's SKILL.md file",
    input_schema: {
      type: "object" as const,
      properties: {
        skill_path: { type: "string", description: "Path to the skill directory" }
      },
      required: ["skill_path"]
    }
  },
  {
    name: "validate_skill",
    description: "Validate a skill against blueprint-lib schema",
    input_schema: {
      type: "object" as const,
      properties: {
        skill_path: { type: "string", description: "Path to the skill directory" }
      },
      required: ["skill_path"]
    },
    cache_control: { type: "ephemeral" as const }  // Breakpoint 1: tools
  }
];

const response = await client.messages.create({
  model: "claude-sonnet-4-5-20250929",
  max_tokens: 4096,
  tools: blueprintTools,
  system: [
    {
      type: "text",
      text: engineEntrypoint,
      cache_control: { type: "ephemeral" }  // Breakpoint 2: engine
    },
    {
      type: "text",
      text: patternContent,
      cache_control: { type: "ephemeral" }  // Breakpoint 3: patterns
    }
  ],
  messages: [
    {
      role: "user",
      content: [
        {
          type: "text",
          text: "Analyze the bp-skill-create skill and suggest improvements.",
          cache_control: { type: "ephemeral" }  // Breakpoint 4: conversation
        }
      ]
    }
  ]
});

console.log('Cache write:', response.usage.cache_creation_input_tokens);
console.log('Cache read:', response.usage.cache_read_input_tokens);
console.log('Uncached:', response.usage.input_tokens);
```

### Breakpoint Strategy for Blueprint-Lib

Using all 4 available breakpoints, ordered by stability:

| Breakpoint | Content | Stability | Why |
|:----------:|---------|-----------|-----|
| 1 | Tool definitions | Very high | Tools rarely change between calls |
| 2 | Engine entrypoint + core instructions | Very high | Changes only on framework releases |
| 3 | Pattern/template files (per-phase) | High | Changes when patterns are updated |
| 4 | Conversation history (latest message) | Low | New content each turn, enables incremental caching |

## 7. Cache Invalidation

### Hierarchy Table

| What Changes | Tools Cache | System Cache | Messages Cache |
|-------------|:-----------:|:------------:|:--------------:|
| Tool definitions (names, descriptions, parameters) | Invalidated | Invalidated | Invalidated |
| Web search toggle | OK | Invalidated | Invalidated |
| Citations toggle | OK | Invalidated | Invalidated |
| Speed setting (fast mode toggle) | OK | Invalidated | Invalidated |
| `tool_choice` parameter | OK | OK | Invalidated |
| Adding/removing images | OK | OK | Invalidated |
| Thinking parameters (enable/disable, budget) | OK | OK | Invalidated |
| Non-tool results with extended thinking | OK | OK | Invalidated (thinking blocks stripped) |

### Blueprint-Specific Invalidation Risks

| Risk | Cause | Mitigation |
|------|-------|------------|
| Pattern file edit during development | Content hash changes | Accept cache miss during authoring; cache restores on next identical call |
| Template placeholder differences | Templates rendered with different values | Cache the raw template, render in messages |
| Tool definition reordering | JSON key order matters | Ensure deterministic serialization |
| Adding a new tool | Tools cache invalidated | Add tools in a stable, sorted order |

## 8. Strategy for Blueprint-Lib

### Mapping the 4 Breakpoints to Framework Layers

```
┌─────────────────────────────────────┐
│  Breakpoint 1: Tool Definitions     │  Changes: ~never (per agent build)
│  - read_file, write_file, validate  │  TTL: 5m (sufficient)
├─────────────────────────────────────┤
│  Breakpoint 2: Engine Entrypoint    │  Changes: per release
│  - Core instructions                │  TTL: 1h (if API agent runs >5m)
│  - Framework version                │
├─────────────────────────────────────┤
│  Breakpoint 3: Phase Context        │  Changes: per phase transition
│  - Relevant pattern files           │  TTL: 5m (sufficient within session)
│  - Relevant templates               │
├─────────────────────────────────────┤
│  Breakpoint 4: Conversation         │  Changes: every turn
│  - User messages + assistant turns  │  TTL: 5m (refreshed each turn)
│  - Tool use/result blocks           │
└─────────────────────────────────────┘
```

### TTL Recommendations per Layer

| Layer | Recommended TTL | Rationale |
|-------|:---------:|-----------|
| Tool definitions | 5m | Refreshed every turn anyway; 5m is sufficient |
| Engine entrypoint | 1h (API agents) / 5m (Claude Code) | API agents may have gaps between calls >5m; Claude Code handles this automatically |
| Phase context | 5m | Active sessions refresh within 5m |
| Conversation | 5m | Each turn refreshes the cache |

### Concurrent Request Considerations

A cache entry only becomes available **after the first response begins**. For parallel agent architectures:

1. Send the first request to populate the cache
2. Wait for the response to start (not complete — just start)
3. Then send parallel follow-up requests that will hit the cache

## 9. Optimization Recommendations

### Progressive Disclosure

Load only the context needed for the current phase:

```
Phase 1 - Discovery:
  system: [engine_entrypoint]                          # ~2K tokens cached

Phase 2 - Analysis:
  system: [engine_entrypoint, analysis_pattern]        # engine from cache + ~5K new

Phase 3 - Generation:
  system: [engine_entrypoint, generation_pattern,      # engine from cache
           template_file]                              # + ~8K new (analysis pattern dropped)

Phase 4 - Validation:
  system: [engine_entrypoint, validation_rules]        # engine from cache + ~4K new
```

**Why this works:** The engine entrypoint is the stable prefix. It gets cached on the first call and read from cache on every subsequent call. Each phase only adds its own specific context.

**Why not load everything at once:** Loading all patterns + templates + validation rules upfront wastes cache writes on content that won't be used until later phases, and may push past the 200K long-context threshold unnecessarily.

### Stable Context Ordering

**Do:**
- Always place engine entrypoint first
- Always place pattern files in a deterministic order (alphabetical, or by phase)
- Use the same tool definitions in the same order across calls

**Don't:**
- Reorder system blocks between calls
- Add/remove tools between calls within the same session
- Include timestamps or request IDs in cached content

### Session Patterns

| Pattern | Description | Cache behavior |
|---------|-------------|----------------|
| **Single-shot** | One request, one response | Cache write only (no read benefit unless repeated) |
| **Multi-turn** | Conversational back-and-forth | Excellent — each turn reads the previous cache and extends it |
| **Agentic loop** | Tool use → result → tool use → result | Excellent — Claude Code handles this automatically; API agents should cache at the conversation end |
| **Parallel fan-out** | One prompt, many variations | Good — seed the cache with first request, then fan out |

## 10. Monitoring & Troubleshooting

### Usage Fields

Every API response includes these fields in the `usage` object:

```json
{
  "usage": {
    "input_tokens": 50,
    "cache_creation_input_tokens": 15000,
    "cache_read_input_tokens": 0,
    "output_tokens": 500
  }
}
```

**Total input tokens** = `cache_read_input_tokens` + `cache_creation_input_tokens` + `input_tokens`

Note: `input_tokens` only counts tokens **after the last cache breakpoint**. It will be much smaller than total input when caching is working.

### With Mixed TTLs

```json
{
  "usage": {
    "input_tokens": 50,
    "cache_creation_input_tokens": 556,
    "cache_read_input_tokens": 100000,
    "output_tokens": 500,
    "cache_creation": {
      "ephemeral_5m_input_tokens": 456,
      "ephemeral_1h_input_tokens": 100
    }
  }
}
```

### Cache Hit Tracking

Monitor these patterns to verify caching is working:

| Request | Expected `cache_creation` | Expected `cache_read` | If not... |
|---------|:------------------------:|:---------------------:|-----------|
| First in session | High | 0 | Normal |
| Second+ in session | Low (new turn only) | High | Cache broken — check invalidation causes |
| After 5m idle | High (re-cache) | 0 | Normal — cache expired |
| After tool definition change | High | 0 | Expected — tools invalidate everything |

### Common Pitfalls

1. **Content below minimum threshold:** If your cached content is under 1,024–4,096 tokens (model-dependent), no caching occurs. Check `cache_creation_input_tokens` — if it's 0 when you expect a write, your content is too short.

2. **Non-deterministic JSON serialization:** Some languages (Swift, Go) randomize JSON key order. Ensure `tool_use` content blocks have stable key ordering.

3. **Invisible system prompt changes:** Toggling web search, citations, or speed setting modifies the system prompt internally, breaking the system cache even if your system content hasn't changed.

4. **Images break message cache:** Adding or removing images anywhere in the prompt invalidates the message cache.

5. **Thinking parameter changes:** Enabling/disabling extended thinking or changing the thinking budget invalidates the message cache (but tools and system caches survive).

6. **More than 20 blocks without breakpoints:** In long agentic sessions, if the conversation exceeds 20 content blocks between cache breakpoints, early blocks may not benefit from caching.

## 11. Advanced Topics

### 1-Hour TTL

Use `"ttl": "1h"` in the `cache_control` block:

```json
"cache_control": {"type": "ephemeral", "ttl": "1h"}
```

**When to use:**
- Agentic side-agents that may take >5m between calls
- Long user conversations where the user may not respond within 5m
- When latency matters and you can't tolerate a cache miss after 5m
- To improve rate limit utilization (cache hits don't count against rate limits)

**When NOT to use:**
- Prompts used more frequently than every 5m — the free refresh on 5m cache is more cost-effective
- The 1-hour write cost is 2x base input (vs 1.25x for 5m)

### Mixed TTLs

You can combine both TTLs in one request, but longer TTL entries **must come first**:

```python
system=[
    {
        "type": "text",
        "text": engine_entrypoint,
        "cache_control": {"type": "ephemeral", "ttl": "1h"}  # Long-lived
    },
    {
        "type": "text",
        "text": session_context,
        "cache_control": {"type": "ephemeral", "ttl": "5m"}  # Short-lived
    },
]
```

Billing uses three positions:
- **Position A:** Highest cache hit (or 0 if none) → charged as cache read
- **Position B:** Highest 1h `cache_control` after A → charged as 1h cache write for (B - A) tokens
- **Position C:** Last `cache_control` block → charged as 5m cache write for (C - B) tokens

### Batch API Usage

Prompt caching works with the Batch API, but cache hits are **best-effort** because requests process concurrently in any order.

**Recommended pattern:**
1. Gather requests sharing a common prefix
2. Send a single-request batch with the shared prefix and a **1-hour cache block**
3. Wait for it to complete (monitor the job)
4. Submit the remaining requests — they'll hit the cache

This is more effective than 5m cache because batch jobs commonly take 5m–1h to complete.

### Workspace-Level Isolation (February 2026 Change)

**Effective February 5, 2026:** Cache isolation changed from **organization-level** to **workspace-level**.

| Platform | Isolation Level |
|----------|----------------|
| Claude API (1P) | Workspace |
| Azure | Workspace |
| Amazon Bedrock | Organization |
| Google Vertex AI | Organization |

**Impact:** If you use multiple workspaces within the same organization, the same prompt sent from different workspaces will **not** share a cache. Review your workspace structure if you rely on cross-workspace cache sharing.

### Concurrent Request Behavior

A cache entry only becomes available **after the first response begins** (not after it completes). For parallel request patterns:

```python
# WRONG: All requests miss cache
responses = [client.messages.create(...) for _ in range(10)]  # All fire simultaneously

# RIGHT: First request seeds cache, rest hit it
first = client.messages.create(...)  # Creates cache entry
# Cache is available as soon as first response begins streaming
rest = [client.messages.create(...) for _ in range(9)]  # Hit cache
```

### Caching with Extended Thinking

- Thinking blocks **cannot** be marked directly with `cache_control`
- However, thinking blocks **are** cached alongside other content in subsequent API calls (e.g., when passing tool results back)
- When cached, thinking blocks count as input tokens in usage metrics
- Cache remains valid when only tool results are passed as user messages
- Cache is invalidated when non-tool-result user content is added (all previous thinking blocks get stripped)

### What Can Be Cached

| Content Type | Cacheable? |
|-------------|:----------:|
| Tool definitions (`tools` array) | Yes |
| System message content blocks | Yes |
| Text messages (user + assistant) | Yes |
| Images & documents in messages | Yes |
| Tool use and tool result blocks | Yes |
| Thinking blocks (direct `cache_control`) | No |
| Thinking blocks (as part of cached conversation) | Yes |
| Citations sub-blocks (direct) | No |
| Citations source documents (top-level) | Yes |
| Empty text blocks | No |
