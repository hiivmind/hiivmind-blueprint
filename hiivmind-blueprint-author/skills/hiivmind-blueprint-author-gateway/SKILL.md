# Generate Gateway Command

Create a gateway command with 3VL intent detection for routing user requests to the appropriate skill.

---

## Overview

This skill generates:
1. A gateway command (`commands/{plugin-name}.md`)
2. Intent mapping configuration (`commands/{plugin-name}/intent-mapping.yaml`)
3. Gateway workflow (`commands/{plugin-name}/workflow.yaml`)

---

## When to Use a Gateway

| Plugin Type | Recommendation |
|-------------|----------------|
| Single skill | No gateway needed |
| 2-3 skills | Optional, simple routing |
| 4+ skills | Gateway with 3VL intent detection |

---

## Phase 1: Analyze Plugin

### Step 1.1: Discover Skills

Scan the plugin for skills:

```
skills = Glob("skills/*/SKILL.md")
```

For each skill, extract:
- Name (from frontmatter)
- Description (trigger keywords)
- Type (prose or workflow)

### Step 1.2: Extract Keywords

From each skill's description, extract trigger keywords:

```
for skill in skills:
  # Parse description frontmatter
  description = skill.frontmatter.description

  # Extract quoted phrases and keyword lists
  keywords = extract_keywords(description)

  skill.keywords = keywords
```

**Keyword extraction patterns:**
- Quoted phrases: `"create corpus"`, `"add source"`
- Trigger phrases: `Triggers on "X", "Y", "Z"`
- Action words: verbs followed by nouns

### Step 1.3: Detect Keyword Overlap

Check for keywords that appear in multiple skills:

```
overlap = {}
for keyword in all_keywords:
  skills_with_keyword = [s for s in skills if keyword in s.keywords]
  if len(skills_with_keyword) > 1:
    overlap[keyword] = skills_with_keyword
```

If significant overlap exists, 3VL intent detection is recommended.

## Phase 2: Design Intent Detection

### Step 2.1: Define Flag Categories

Group keywords into flag categories:

```yaml
intent_flags:
  has_init:
    keywords: ["create", "new", "initialize", "init"]
    description: "User wants to create something new"

  has_modify:
    keywords: ["add", "update", "edit", "change"]
    description: "User wants to modify existing"

  has_query:
    keywords: ["find", "search", "show", "list", "what"]
    description: "User is asking a question"

  has_help:
    keywords: ["help", "how do i", "?"]
    description: "User needs guidance"

  # Skill-specific flags
  has_{skill_name}:
    keywords: [skill-specific keywords]
    description: "Specific to {skill_name}"
```

### Step 2.2: Create Intent Rules

Map flag combinations to skills:

```yaml
intent_rules:
  # Pure single intents (highest priority)
  - name: "init_only"
    conditions:
      has_init: T
      has_modify: F
      has_query: F
    action: delegate_init
    priority: 90
    description: "Create new {thing}"

  # Compound intents
  - name: "help_with_init"
    conditions:
      has_help: T
      has_init: T
    action: delegate_init
    priority: 80
    description: "Help me create..."

  # Query intents
  - name: "discover_status"
    conditions:
      has_query: T
      has_init: F
    action: delegate_discover
    priority: 85
    description: "Show available {things}"

  # Fallback
  - name: "show_menu"
    conditions: {}
    action: show_main_menu
    priority: 10
    description: "No clear intent, show menu"
```

## Phase 3: Generate Gateway Command

### Step 3.1: Create Command Directory

```bash
mkdir -p commands/{plugin_name}
```

### Step 3.2: Generate intent-mapping.yaml

Write the intent detection configuration:

```yaml
# Intent mapping for {plugin_name} gateway
# Generated by hiivmind-blueprint-gateway

intent_flags:
  {for each flag}
  {flag_name}:
    keywords:
      {for each keyword}
      - "{keyword}"
      {/for}
    {if negative_keywords}
    negative_keywords:
      {for each neg_keyword}
      - "{neg_keyword}"
      {/for}
    {/if}
    description: "{description}"
  {/for}

intent_rules:
  {for each rule}
  - name: "{rule_name}"
    conditions:
      {for each condition}
      {flag}: {value}
      {/for}
    action: {action}
    priority: {priority}
    description: "{description}"
  {/for}
```

### Step 3.3: Generate Gateway workflow.yaml

Write the gateway workflow:

```yaml
name: "{plugin_name}-gateway"
version: "1.0.0"
description: >
  Gateway command for {plugin_name}. Routes requests to appropriate skills
  using 3VL intent detection.

entry_preconditions: []

initial_state:
  phase: "detect"
  arguments: null
  intent: null
  flags:
    has_arguments: false
  computed: {}

start_node: check_arguments

nodes:
  check_arguments:
    type: conditional
    description: "Check if user provided arguments"
    condition:
      type: state_not_null
      field: arguments
    branches:
      true: parse_intent
      false: show_main_menu

  parse_intent:
    type: action
    description: "Parse user input with 3VL flags"
    actions:
      - type: parse_intent_flags
        input: "${arguments}"
        flag_definitions: "${intent_flags}"
        store_as: computed.intent_flags
      - type: match_3vl_rules
        flags: "${computed.intent_flags}"
        rules: "${intent_rules}"
        store_as: computed.intent_matches
    on_success: check_clear_winner
    on_failure: show_main_menu

  check_clear_winner:
    type: conditional
    description: "Check if we have a clear intent match"
    condition:
      type: flag_set
      flag: computed.intent_matches.clear_winner
    branches:
      true: execute_matched_intent
      false: show_disambiguation

  execute_matched_intent:
    type: action
    description: "Execute the matched intent's action"
    actions:
      - type: dynamic_route
        action: "${computed.intent_matches.winner.action}"
    on_success: "${computed.dynamic_target}"
    on_failure: show_main_menu

  show_disambiguation:
    type: user_prompt
    prompt:
      question: "I found multiple possible matches. Which did you mean?"
      header: "Clarify"
      options:
        # Dynamically built from top_candidates
    on_response:
      # Dynamic handlers for each candidate

  show_main_menu:
    type: user_prompt
    prompt:
      question: "What would you like to do with {plugin_name}?"
      header: "Menu"
      options:
        {for each skill}
        - id: {skill_id}
          label: "{skill_label}"
          description: "{skill_short_desc}"
        {/for}
    on_response:
      {for each skill}
      {skill_id}:
        consequence:
          - type: set_state
            field: intent
            value: {skill_id}
        next_node: delegate_{skill_id}
      {/for}

  # Delegation nodes for each skill
  {for each skill}
  delegate_{skill_id}:
    type: action
    description: "Delegate to {skill_name}"
    actions:
      - type: invoke_skill
        skill: "{skill_name}"
        args: "${arguments}"
    on_success: success
    on_failure: error_delegation
  {/for}

endings:
  success:
    type: success
    message: "Request handled by {plugin_name}"

  error_delegation:
    type: error
    message: "Failed to delegate to skill"
```

## Intent Detection

This gateway uses 3VL (3-valued logic) intent detection with Kleene semantics (v2.1.0):

### Flag Values
- **T (True)**: Keyword matched
- **F (False)**: Negative keyword matched
- **U (Unknown)**: No match

### Rule Matching (Kleene Logic)
- **Rule U** = "don't care" (wildcard) - condition is skipped entirely
- **State U vs Rule T/F** = soft match (uncertain satisfaction)
- **Hard match**: state T + rule T, or state F + rule F
- **Exclusion**: state T + rule F, or state F + rule T (candidate rejected)

### Ranking Algorithm
Candidates are ranked by: `(-hard_matches, +soft_matches, +effective_conditions)`
1. Most hard matches wins (definite matches)
2. Fewest soft matches wins (less uncertainty)
3. Fewest effective conditions wins (more specific rule)

See `docs/intent-detection-guide.md` for usage details.
